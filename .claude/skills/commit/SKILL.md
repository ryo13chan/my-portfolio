# commit

このプロジェクトの Git 運用ルールに従って、ブランチ作成とコミットを行う。

## 鉄則

- **`main` ブランチには絶対に直接コミットしない。** 現在のブランチが `main` なら、必ず先に新しいブランチを切る。
- 既に作業ブランチ（`main` 以外）にいる場合は、そのブランチにコミットしてよい。

## 手順

1. 現在の状態を確認する:
   - `git branch --show-current` で現在のブランチを確認
   - `git status` と `git diff`（必要なら `git diff --staged`）で変更内容を把握

2. 現在のブランチが `main` の場合、変更内容に応じた名前で新しいブランチを切る:
   - 接頭辞: 新機能 `feat/`、バグ修正 `fix/`、雑務・設定 `chore/`、ドキュメント `docs/`
   - ブランチ名は変更内容を表す簡潔な英語のケバブケースにする。
   - 末尾に `_MMDD`（作成日の月日。`date +%m%d` で取得）を付ける。
   - 例: `git switch -c feat/hero-section_$(date +%m%d)` → `feat/hero-section_0613`

3. 変更をステージしてコミットする:
   - **無理に1コミットにまとめない。** まず変更全体を見渡し、関心ごとが異なるものは論理単位ごとに分割する（例: 機能追加・リファクタ・ドキュメント更新は別コミット）。
   - 各単位ごとに関連ファイルだけを `git add <path>...` でステージし、個別にコミットする（全部まとめる `git add -A` は単一の関心ごとのときだけ）。
   - 各コミットは単独で意味が通る最小単位にし、メッセージは変更内容を簡潔に表す。本文末尾に必ず次のトレーラーを付ける:
     ```
     Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
     ```

4. コミット後、`git log --oneline -1` で結果を報告する。プッシュはユーザーが明示的に頼んだときのみ行う。

## 注意

- 技術構成（依存パッケージ・Node バージョン等）に変更があった場合は、コミット前に [TECH_STACK.md](../../../TECH_STACK.md) が更新されているか確認する。
